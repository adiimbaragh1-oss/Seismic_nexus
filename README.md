# Seismic Nexus — النسخة الكاملة (Backend + Frontend)

مختبر محاكاة لاختبار فرضية دمج عوامل فلكية وجيوفيزيائية ومناخية للتنبؤ بالزلازل.
**تنويه مهم:** لا يوجد إجماع علمي يثبت أن اصطفاف الكواكب يؤثر على الزلازل. هذه أداة
تعليمية/استكشافية تستخدم بيانات حقيقية (USGS و Open-Meteo) ضمن معادلة افتراضية، وليست
أداة تنبؤ زلزالي علمية.

## البنية

```
seismic-nexus-app/
  backend/          Node.js + Express + SQLite — الـ API، تسجيل الدخول، المحرك الخلفي، Telegram
    server.js       نقطة الدخول
    db.js           قاعدة البيانات (SQLite)
    auth.js         تسجيل الدخول (bcrypt + JWT عبر كوكي httpOnly)
    telegram.js     إرسال تنبيهات Telegram
    riskEngine.js   المعادلة الأساسية + جلب بيانات USGS/Open-Meteo/الفلك
    package.json
    .env.example    انسخه إلى .env واملأه
  frontend/         الواجهة (HTML/CSS/JS خالص، بدون build step)
    index.html
    style.css
    script.js
```

## التشغيل خطوة بخطوة

### المتطلبات
- Node.js نسخة 18 أو أحدث (يحتاج `fetch` المدمج في Node)

### 1) تثبيت الاعتماديات
```bash
cd backend
npm install
```

### 2) إعداد المتغيرات البيئية
```bash
cp .env.example .env
```
افتح `.env` وعدّل:
- `JWT_SECRET`: أي نص عشوائي طويل
- `TELEGRAM_BOT_TOKEN` و `TELEGRAM_CHAT_ID`: راجع قسم Telegram أدناه (اختياري، يعمل الموقع بدونهما لكن بدون تنبيهات)

### 3) تشغيل الخادم
```bash
npm start
```
سيطبع الخادم:
```
✅ Seismic Nexus API يعمل على http://localhost:4000
```
افتح المتصفح على **http://localhost:4000** — الخادم يقدّم الواجهة الأمامية والـ API معاً من نفس المنفذ.

> عند أول تشغيل، يحتاج الخادم بضع ثوانٍ لجلب البؤر من USGS وبيانات الطقس من Open-Meteo قبل أن تمتلئ اللوحة بالبيانات.

## إعداد تنبيهات Telegram

1. تحدّث إلى [@BotFather](https://t.me/BotFather) في تيليجرام وأرسل `/newbot` واتبع التعليمات للحصول على **التوكن**
2. ابدأ محادثة مع بوتك الجديد وأرسل له أي رسالة (مثلاً "hi")
3. افتح هذا الرابط في المتصفح (استبدل `<TOKEN>`):
   `https://api.telegram.org/bot<TOKEN>/getUpdates`
4. ابحث في الاستجابة عن `"chat":{"id": ...}` وانسخ الرقم — هذا هو `TELEGRAM_CHAT_ID`
5. ضع القيمتين في ملف `.env` وأعد تشغيل الخادم

بعد الإعداد، سيرسل الخادم تلقائياً:
- تنبيهاً عند عبور مؤشر الخطر المحسوب لأي بؤرة إلى مستوى "متوسط" أو "حرج"
- تنبيهاً فورياً عند تسجيل USGS لزلزال حقيقي جديد بالقرب من إحدى البؤر

يمكنك اختبار الاتصال بعد تسجيل الدخول عبر:
```bash
curl -X POST http://localhost:4000/api/alerts/test -H "Cookie: sn_token=<التوكن من المتصفح>"
```
أو ببساطة سجّل الدخول من الواجهة ونفّذ طلب POST إلى `/api/alerts/test` من أدوات المطوّر.

## تسجيل الدخول

- أي زائر يرى اللوحة والخريطة والرسوم البيانية دون تسجيل دخول (قراءة فقط)
- **تعديل أوزان العوامل** يتطلب تسجيل الدخول (الأوزان مشتركة بين كل الزوار بعد الحفظ)
- أنشئ حساباً من زر "تسجيل الدخول" أعلى الصفحة → تبويب "حساب جديد"
- كلمات المرور مشفّرة بـ bcrypt، والجلسة محفوظة بكوكي httpOnly موقّع بـ JWT (صالح 7 أيام)

## قاعدة البيانات ونظام التحليل التاريخي

تُستخدم **SQLite** (ملف `backend/seismic.db` يُنشأ تلقائياً — لا حاجة لتثبيت خادم قاعدة بيانات).

- كل دقيقة، يحسب المحرك الخلفي مؤشر الخطر لكل بؤرة ويحفظه في جدول `readings` (الوقت، والعوامل الأربعة، والمؤشر النهائي)
- استخدم `GET /api/hotspots/:id/history?hours=168` لاسترجاع تاريخ أي بؤرة (حتى 30 يوماً)، وهو ما يغذّي الرسم البياني الزمني في الواجهة
- جدول `alerts` يحتفظ بسجل دائم لكل التنبيهات، مع علامة `telegram_sent` توضح هل وصل التنبيه فعلاً

### الانتقال إلى PostgreSQL (اختياري)
الكود مكتوب بحيث يسهل الاستبدال:
1. ثبّت مكتبة `pg` بدل `better-sqlite3`
2. استبدل `backend/db.js` بإعداد `Pool` من `pg` وحوّل الاستعلامات المتزامنة إلى `async/await`
3. بدّل `INTEGER PRIMARY KEY AUTOINCREMENT` بـ `SERIAL PRIMARY KEY` في تعريفات الجداول
4. بقية الكود (`server.js`, `riskEngine.js`) يبقى شبه كما هو لأن الاستعلامات تستخدم SQL قياسي

## واجهة الـ API الخاصة بك (ملخص)

| Method | Path | وصف | يتطلب تسجيل دخول |
|---|---|---|---|
| POST | `/api/auth/register` | إنشاء حساب | لا |
| POST | `/api/auth/login` | تسجيل الدخول | لا |
| POST | `/api/auth/logout` | تسجيل الخروج | لا |
| GET | `/api/auth/me` | معرفة المستخدم الحالي | لا |
| GET | `/api/hotspots` | كل البؤر + الخطر المحسوب حالياً | لا |
| GET | `/api/hotspots/:id/history?hours=168` | تاريخ بؤرة معينة | لا |
| GET | `/api/weights` | الأوزان الحالية | لا |
| PUT | `/api/weights` | تعديل الأوزان | **نعم** |
| GET | `/api/alerts?limit=30` | آخر التنبيهات | لا |
| POST | `/api/alerts/test` | إرسال رسالة اختبار لـ Telegram | **نعم** |

## ملاحظات أمان قبل النشر الفعلي (Production)

- غيّر `JWT_SECRET` إلى قيمة عشوائية قوية ولا تُشاركها
- شغّل الخادم خلف HTTPS (سيؤدي `NODE_ENV=production` إلى تفعيل كوكي `secure`)
- فكّر بإضافة تحديد لعدد المحاولات (rate limiting) على `/api/auth/login` لمنع التخمين العشوائي لكلمات المرور
- هذا مشروع تعليمي/عرض توضيحي وليس منتجاً جاهزاً للإنذار الحقيقي بالزلازل
